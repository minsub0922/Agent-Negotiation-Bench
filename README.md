# negmas-based N-Agent Travel Itinerary Negotiation Project

This project is an experimental pipeline that uses `negmas` to negotiate travel itineraries based on each agent's calendar and preferences.

- Generate random scenario datasets
- Generate negotiation utterances with local LLMs (per-agent model paths supported)
- Save detailed negotiation logs (JSONL) plus human-readable dialogue JSON
- Save a human-only transcript that contains only chat messages
- Save quantitative evaluation metrics plus readable MD/JSON summaries

## 1) Installation

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

If you also want to use teacher models through vLLM:

```bash
pip install -r requirements-vllm.txt
```

## 2) Generate Dataset

```bash
python -m src.main generate-dataset \
  --num-scenarios 30 \
  --num-agents 3 \
  --seed 42 \
  --horizon-days 14 \
  --dataset-id ds_paper_v1 \
  --output data/scenarios.jsonl
```

## 3) Run Negotiation (Using an Existing Dataset)

### 3-1. Use local HF models (per-agent model paths)

```bash
python -m src.main run \
  --dataset data/scenarios.jsonl \
  --dataset-id ds_paper_v1 \
  --run-id run_modelA_modelB_ds1 \
  --llm-backend hf \
  --num-gpus 2 \
  --torch-dtype bfloat16 \
  --decision-policy llm-hybrid \
  --require-explicit-accept \
  --agent-model-paths agent_a=/absolute/path/to/model_a,agent_b=/absolute/path/to/model_b,agent_c=/absolute/path/to/model_c \
  --llm-max-new-tokens 256 \
  --max-steps 40 \
  --output-dir outputs
```

### 3-2. Use one model path for all agents

```bash
python -m src.main run \
  --dataset data/scenarios.jsonl \
  --dataset-id ds_paper_v1 \
  --llm-backend hf \
  --num-gpus 1 \
  --torch-dtype bfloat16 \
  --decision-policy llm-hybrid \
  --require-explicit-accept \
  --model-path /absolute/path/to/model \
  --llm-max-new-tokens 256 \
  --max-steps 40 \

```

### 3-3. Dummy LLM for quick checks

## Output Structure

Outputs are stored under `outputs/<run_id>/` when you run an experiment. (`run_id` is auto-generated by default with model signature + dataset id.)

- `experiment_summary.csv`: Per-scenario row-level quantitative data
- `experiment_summary.json`: Run-level aggregates (means, agreement rate, etc.)
- `my_metrics.csv`: Flattened per-scenario rows from the new `my_metrics` block
- `my_metrics_summary.json`: Per-run/per-scenario mean and std summaries from `my_metrics`
- `quant_summary.md`: Human-readable quantitative summary report
- `quant_summary_human_readable.json`: Quant summary with formatted strings (percentages, etc.)
- `github_issue_collection.md`: Per-scenario Issue file list + experiment overview
- `run_config.json`: Run parameters (model paths, seed, backend, etc.)
- `dataset_config_snapshot.json`: Snapshot of dataset identifiers used for the run (id/hash/path)
- `scenario_xxxx/scenario.json`: Original scenario payload
- `scenario_xxxx/metrics.json`: Negotiation metrics for that scenario
  - Existing metrics remain unchanged, and a `my_metrics` block is added at the same level.
- `scenario_xxxx/metrics_human_readable.md`: Per-scenario quantitative summary
- `scenario_xxxx/negmas_trace.jsonl`: `negmas`-level negotiation trace
- `scenario_xxxx/agent_event_log.jsonl`: Detailed log including propose/respond events and LLM prompt/raw outputs
  - The `decision_source`, `llm_action`, `llm_choice`, and `offer_candidates` fields let you track decision provenance.
- `scenario_xxxx/dialogue_human_readable.json`: Human-readable dialogue log + per-turn utilities
- `scenario_xxxx/chat_transcript.json`: Chat-style log containing messages only (JSON)
- `scenario_xxxx/chat_transcript.txt`: Chat-style log containing messages only (TXT)
- `scenario_xxxx/issue_bundle.json`: Combined structure with chat + quantitative + detailed JSON
- `scenario_xxxx/issue_bundle.md`: Combined markdown ready to use as a GitHub Issue body

## Main Quantitative Metrics

- `agreement_reached`: Whether agreement was reached
- `utility_by_agent`: Per-agent utility under the final agreement
- `reservation_by_agent`: Per-agent reservation value
- `calendar_conflict_ratio_by_agent`: Per-agent calendar conflict ratio
- `my_metrics`: Additional evaluation block computed independently from legacy metrics
- `social_welfare`: Sum of utilities over all agents
- `nash_product`: Nash product based on `Π max(u_i-r_i, 0)` over all agents
- `pareto_optimal`: Whether the final agreement is Pareto optimal
- `welfare_ratio_vs_best`, `nash_ratio_vs_best`: Ratios against the optimum in full outcome space
- Backward-compatible keys (`utility_agent_a`, `utility_agent_b`, etc.) are still stored

For metric interpretation, see `docs/QUANT_METRICS_GUIDE.md`.

## Notes

- LLM responses are used for dialogue generation, while proposal/acceptance decisions are handled by `negmas` negotiator strategy (`LLMCalendarNegotiator`) and utility functions.
- To reduce truncation, the default `--llm-max-new-tokens` was increased to 256, and hard length clipping was removed from output post-processing.
- If local model memory usage is high, use `--allow-dummy-fallback` for sanity checks.
- In `run`/`full`, auto-generated `run_id` includes model signature + dataset id (or set it explicitly with `--run-id`).
- During dataset generation, `<dataset>.config.json` is saved with `dataset_id`/hash/seed information.
- If you set `--num-gpus`, agent placement is assigned automatically.
  - `--num-gpus 0`: all agents on CPU
  - `--num-gpus 1`: all agents on `cuda:0`
  - `--num-gpus >= 2`: round-robin placement `agent_i -> cuda:(i % num_gpus)`
  - If all agents use the same model path (`--model-path` or identical `--agent-model-paths`), model instances are shared to reduce memory usage.
- Actual placement is recorded in run logs under `[PLACEMENT] ...` and in `run_config.json > agent_placement`.
- If a 14B model is slow, lower `--llm-max-new-tokens` to `64` or `96` first.
- If you hit OOM, stabilize first with `--torch-dtype bfloat16` (recommended on H100), `--llm-max-new-tokens 64`, and `--max-steps 20`.
- Default `--decision-policy` is `llm-hybrid`, so model output influences proposal choice and accept/reject decisions.
  - `heuristic`: utility-based rules only
  - `llm-hybrid`: LLM output + minimal guardrails
  - `llm-only`: primarily model-driven decisions
- Default `--require-explicit-accept` is `True`. If the model does not explicitly output `ACTION: ACCEPT`, it is not treated as agreement.

## Synthetic Trace -> TRL Training -> Evaluation Pipeline

The following capabilities were added.

- `generate-traces`: Generate diverse negotiation traces with teacher models (vLLM/HF/dummy) and automatically extract SFT/DPO training data
- `build-training-data`: Rebuild only SFT/DPO training data from existing run outputs
- `train-sft`: Target-model supervised fine-tuning with TRL `SFTTrainer`
- `train-rl`: Preference optimization with TRL `DPOTrainer` (RLHF-style)
- `evaluate-models`: Batch-evaluate multiple negotiator models on the same dataset and build a leaderboard

### 1) Generate Teacher Traces + Build Training Data

```bash
python -m src.main generate-traces \
  --dataset data/scenarios.jsonl \
  --llm-backend vllm \
  --teacher-model-paths teacher_a=/models/teacher_a,teacher_b=/models/teacher_b \
  --episodes-per-teacher 3 \
  --temperature 0.8 \
  --temperature-jitter 0.25 \
  --top-p 0.92 \
  --top-p-jitter 0.05 \
  --shuffle-agent-order \
  --shuffle-scenario-order \
  --output-dir outputs/synthetic \
  --trace-id teacher_mix_v1
```

Generated outputs:

- `outputs/synthetic/<trace_id>/runs/*`: Per-episode negotiation execution artifacts
- `outputs/synthetic/<trace_id>/training_data/sft_{train,eval}.jsonl`
- `outputs/synthetic/<trace_id>/training_data/dpo_{train,eval}.jsonl`
- `outputs/synthetic/<trace_id>/trace_generation_summary.{json,csv}`

### 2) SFT Training

```bash
python -m src.main train-sft \
  --model-path /models/target_base \
  --train-file outputs/synthetic/teacher_mix_v1/training_data/sft_train.jsonl \
  --eval-file outputs/synthetic/teacher_mix_v1/training_data/sft_eval.jsonl \
  --output-dir outputs/train/target_sft_v1 \
  --max-seq-length 2048 \
  --gradient-accumulation-steps 8 \
  --learning-rate 2e-5 \
  --num-train-epochs 1.5 \
  --bf16 \
  --use-lora
```

### 3) RL (DPO) Stage

```bash
python -m src.main train-rl \
  --model-path outputs/train/target_sft_v1/final_model \
  --ref-model-path /models/target_base \
  --train-file outputs/synthetic/teacher_mix_v1/training_data/dpo_train.jsonl \
  --eval-file outputs/synthetic/teacher_mix_v1/training_data/dpo_eval.jsonl \
  --output-dir outputs/train/target_dpo_v1 \
  --beta 0.1 \
  --max-seq-length 2048 \
  --max-prompt-length 1024 \
  --gradient-accumulation-steps 8 \
  --learning-rate 5e-6 \
  --num-train-epochs 1.0 \
  --bf16 \
  --use-lora
```

### 4) Model Evaluation + Leaderboard

```bash
python -m src.main evaluate-models \
  --dataset data/scenarios.jsonl \
  --llm-backend vllm \
  --model-paths base=/models/target_base,sft=outputs/train/target_sft_v1/final_model,rl=outputs/train/target_dpo_v1/final_model \
  --num-scenarios 100 \
  --output-dir outputs/evaluations \
  --eval-id negotiator_eval_v1
```

Generated outputs:

- `outputs/evaluations/<eval_id>/runs/*`: Per-model detailed negotiation logs and quantitative metrics
- `outputs/evaluations/<eval_id>/leaderboard.{json,csv,md}`

For detailed pipeline behavior, see `docs/SYNTHETIC_NEGOTIATION_TRAINING_PIPELINE.md`.
